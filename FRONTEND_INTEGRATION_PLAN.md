# FRONTEND INTEGRATION PLAN

This plan maps exactly where the FastAPI backend response payload will inject into the React architecture, and what changes are required.

## 1. API Layer Modifications (`src/services/api.js`)
Currently, `api.js` returns an object structured as:
```javascript
{
  intent: { category, occasion, budget, interest, volume },
  shoppingPlan: [ { id, name, price, rating, whyWePickedThis, imageColor } ]
}
```

Our Backend exposes `POST /mission` returning:
```javascript
{
  mission: { ... },
  recommendations: [ ... ],
  bundle: { readiness_score, remaining_budget, ... },
  analytics: { confidence, shopping_strategy, ... },
  summary: "...",
  shopping_tips: [ ... ]
}
```

**Integration Action:** 
We will update `postIntent` to set `USE_MOCK_API = false`, hit `http://localhost:8000/api/mission`, and map the backend response keys directly to the state objects expected by the frontend components.

## 2. Component Integration Mapping

### A. Mission Input
- **Component:** `FloatingCopilot.jsx` / `ConsumerView.jsx` (Chat Input)
- **Integration:** The raw text input typed by the user will be passed directly into `postIntent(query)`, which will `POST` to our FastAPI backend.

### B. Loading States
- **Component:** `ConsumerView.jsx`
- **Integration:** Remove the artificial `setTimeout` in `api.js`. `ConsumerView` already utilizes an `isGenerating` state hook. This boolean flag will natively track the ~3.0s latency required by the backend to fetch Neo4j and Gemini responses, triggering the skeleton UI animations.

### C. Recommendation Cards
- **Component:** `ConsumerView.jsx` (List of cards rendered from `shoppingPlan`)
- **Integration:** Map `backendResponse.recommendations` to `shoppingPlan`. 
  - `recommendation.name` -> Card Title
  - `recommendation.price` -> Card Price
  - `recommendation.gemini_reason` (if available) -> `whyWePickedThis`

### D. Bundle UI (Money Left / Readiness)
- **Component:** `MoneyLeftCard.jsx`
- **Integration:** Pull `backendResponse.bundle.remaining_budget` and `backendResponse.bundle.readiness_score`. Display the backend's calculated readiness percentage instead of hardcoded values.

### E. Analytics UI (Seller Intelligence / Consumer Strategy)
- **Component:** `SignalFeed.jsx`, `IntelligenceLoop.jsx`, and Seller Dashboards
- **Integration:** We will pipe `backendResponse.analytics.shopping_strategy` and `backendResponse.analytics.confidence` directly into the Consumer copilot interface. For the seller view, we will need to ensure the backend exposes a dedicated endpoint or we parse the generated JSON into `postSellerQuery`.

### F. AI Explanation (Summary)
- **Component:** `ConsumerView.jsx` (The chat response bubble)
- **Integration:** We will inject `backendResponse.summary` (The conversational text generated by `ReasoningService`) directly into the agent's chat bubble as the primary response to the user's mission.

## 3. Preparation Steps Before Execution
1. Create a `.env.local` in `d:\frontend` to inject `VITE_API_URL=http://localhost:8000`.
2. Disable `USE_MOCK_API` in `api.js`.
3. Implement a try/catch mapping layer to ensure if the backend returns an empty `recommendations` array (for unsupported missions), the UI gracefully displays the backend's friendly rejection summary.
